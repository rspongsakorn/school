# นำเข้าการชำระเงินย้อนหลังด้วยไฟล์ CSV — Design

วันที่: 2026-06-09

## เป้าหมาย

เพิ่มความสามารถนำเข้า "การชำระเงินย้อนหลัง" จากสมุดบัญชีกระดาษ ผ่านการอัปโหลดไฟล์ CSV
ระบบจะบันทึกการชำระให้แต่ละนักเรียน จัดสรรเงินเข้าใบแจ้งหนี้ที่ค้างอยู่ และออกใบเสร็จ
โดย **เลขใบเสร็จเรียงลำดับตามวันที่ที่ชำระเงิน**

## ขอบเขตและข้อสมมติ (ยืนยันกับผู้ใช้แล้ว)

- **การจับคู่นักเรียน:** CSV มีทั้ง `student_code` และ `student_name` — ใช้รหัสเป็นคีย์จับคู่หลัก, ชื่อใช้แสดงเทียบเพื่อยืนยัน
- **ใบแจ้งหนี้:** นักเรียนมีใบแจ้งหนี้ในระบบอยู่แล้ว การนำเข้าเพียงจัดสรรเงินเข้าใบแจ้งที่ค้าง (FIFO) และออกใบเสร็จ
- **วิธีชำระ:** เป็นเงินสด (`cash`) ทั้งหมด ไม่มีคอลัมน์วิธีชำระใน CSV
- **เลขใบเสร็จ:** ปุ่มนำเข้าทำงานแบบ "ปกติ" คือต่อเลขจาก sequence สูงสุดปัจจุบัน +1 ไล่ตามลำดับวันที่ภายใน batch — ไม่มีการลบข้อมูลฝังในปุ่ม
- **การล้างข้อมูลเดิม:** ทำครั้งเดียวแยกต่างหาก (รันคำสั่ง) ก่อนนำเข้าจริง ไม่อยู่ในฟีเจอร์

## รูปแบบไฟล์ CSV

```csv
student_code,student_name,amount,paid_date
14333,นาลันทา ศรีวัฒนพงศ์,3600,06/05/2569
14399,อลิสา มูลทา,2000,12/05/2569
13742,ณัฐธีรา ตรัยศิลานันท์,6155,22/05/2569
```

| คอลัมน์ | ความหมาย | การใช้งาน |
|---|---|---|
| `student_code` | รหัสนักเรียน | คีย์จับคู่หลัก |
| `student_name` | ชื่อ-สกุล | แสดงเทียบในพรีวิว (เตือนถ้าไม่ตรง แต่ไม่บล็อก) |
| `amount` | ยอดที่ชำระ | จัดสรรเข้าใบแจ้ง FIFO |
| `paid_date` | วันที่ชำระ | รูปแบบ **พ.ศ. DD/MM/YYYY** → แปลงเป็น ค.ศ. (ลบ 543); ใช้ตั้ง `paid_at` และเรียงเลขใบเสร็จ |

## สถาปัตยกรรม (แนวทางที่เลือก: parse ฝั่ง browser + พรีวิว + ยืนยันครั้งเดียว)

### UI

- ปุ่ม **"นำเข้า CSV"** ที่มุมขวาบนของการ์ด "รับชำระเงิน" ในหน้า `/payments`
- กดปุ่ม → เปิด dialog: เลือกไฟล์ + ลิงก์ "ดาวน์โหลดเทมเพลต CSV"
- เลือกไฟล์ → parse ฝั่ง browser → แสดงตารางพรีวิว
- ปุ่ม "ยืนยันนำเข้า" → เรียก server action → แสดงผลสรุป

### ตารางพรีวิว

คอลัมน์: ลำดับ, รหัส, ชื่อใน CSV, ชื่อในระบบ, ยอดชำระ, ยอดค้าง, วันที่, สถานะ

### กฎตรวจสอบแต่ละแถว

| สถานะ | เงื่อนไข | นำเข้า? | สี |
|---|---|---|---|
| ✓ ชำระเต็ม | `amount` = ยอดค้างรวม | ใช่ | เขียว |
| ◐ ชำระบางส่วน | `amount` < ยอดค้างรวม | ใช่ | ฟ้า |
| ✗ ไม่พบรหัส | รหัสไม่มีในระบบ | ข้าม | แดง |
| ✗ ยอดเกินค้าง | `amount` > ยอดค้างรวม | ข้าม | แดง |
| ✗ ไม่มียอดค้าง | ยอดค้าง = 0 (อาจนำเข้าไปแล้ว) | ข้าม | แดง |
| ✗ รูปแบบผิด | วันที่/ยอดผิดรูปแบบ | ข้าม | แดง |
| ⚠ ชื่อไม่ตรง | รหัสตรงแต่ชื่อต่างจากระบบ | ใช่ (เตือน) | เหลือง |

สรุปก่อนยืนยัน: "พร้อมนำเข้า N รายการ · ข้าม M รายการ · ยอดรวม ฿X"
แถวที่ข้ามจะไม่ถูกบันทึก ผู้ใช้แก้ CSV แล้วนำเข้าใหม่เฉพาะที่ตกได้

ชำระบางส่วนรองรับโดยตรง: `allocatePaymentFifo` จัดสรรเท่าที่มี และ `deriveInvoiceStatus`
ตั้งสถานะใบแจ้งเป็น `partial` อัตโนมัติ ยอดค้างที่เหลือคงอยู่ในระบบ

### Backend — server action `importPaymentsBackfill`

action ใหม่ แยกจาก `recordPayment` เดิม (ไม่แก้ของเดิม)

ลำดับการทำงาน:

1. ตรวจสิทธิ์ด้วย `requireFinanceAction()`
2. **เรียงทั้ง batch ตาม `paid_date` จากเก่า→ใหม่** (หัวใจของการเรียงเลขใบเสร็จ)
3. ดึงเลข sequence สูงสุดปัจจุบันครั้งเดียว (`parseMaxSequence`) แล้วไล่ +1 ในหน่วยความจำ
4. วนทีละแถวตามลำดับวันที่:
   - จับคู่นักเรียนจาก `student_code`
   - ดึงใบแจ้งค้างด้วย `getStudentOutstandingInvoices` → `allocatePaymentFifo`
   - insert `payments`: ตั้ง `paid_at` = วันที่ในไฟล์, `payment_method: 'cash'`,
     `recorded_by` = ผู้นำเข้า, `note: 'นำเข้าย้อนหลัง'`, `status: 'active'`
   - insert `payment_allocations`
   - insert `receipts` (snapshot + เลขใบเสร็จไล่ตามลำดับ)
   - update `student_invoices` (paid_amount + status ผ่าน `deriveInvoiceStatus`)
5. คืนผลสรุป: สำเร็จกี่รายการ, ล้มเหลวกี่รายการ พร้อมเหตุผลรายแถว

ประเด็นความถูกต้อง:

- **ตรวจซ้ำฝั่ง server** ไม่เชื่อพรีวิวฝั่ง browser อย่างเดียว (กันไฟล์ถูกแก้/ข้อมูลเปลี่ยน
  ระหว่างพรีวิวกับยืนยัน) — เช็ครหัส + ยอดค้าง + รูปแบบซ้ำ
- **แถวที่ error ไม่ทำให้ทั้ง batch ล้ม** ข้ามเฉพาะแถวนั้น บันทึกที่เหลือต่อ แล้วรายงาน
- **เลขใบเสร็จกันชนกัน** เพราะไล่ sequence ในหน่วยความจำตามลำดับวันที่ จึงไม่ซ้ำและเรียงตามวันที่จริง
- **กันนำเข้าซ้ำ** หากนำเข้าไฟล์เดิมซ้ำ ยอดค้างจะเป็น 0 → ตกกฎ "ไม่มียอดค้าง"/"ยอดเกินค้าง"
  จึงข้ามอัตโนมัติ

## การล้างข้อมูลครั้งเดียว (แยกจากฟีเจอร์)

ก่อนนำเข้าจริง รันคำสั่งแยกครั้งเดียวเพื่อล้างใบเสร็จทดสอบ:
ลบ `receipts` → `payment_allocations` → `payments` แล้วรีเซ็ต `paid_amount`/`status`
ของใบแจ้งกลับเป็น 0/`unpaid` หลังจากนั้นการนำเข้าจะเริ่มที่ `2569/00001`

## ไฟล์ที่เกี่ยวข้อง

- `src/lib/actions/payments.ts` — เพิ่ม action `importPaymentsBackfill`
- `src/lib/finance/csv-import.ts` (ใหม่) — parse + validate CSV, แปลงวันที่ พ.ศ.→ค.ศ.
- `src/components/finance/payment-import-dialog.tsx` (ใหม่) — dialog เลือกไฟล์ + ตารางพรีวิว

- reuse: `allocatePaymentFifo`, `deriveInvoiceStatus`, `formatReceiptNumber`, `parseMaxSequence`,
  `getStudentOutstandingInvoices`
